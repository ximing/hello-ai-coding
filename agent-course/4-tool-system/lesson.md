# 第 4 课：工具系统，让 Agent 真正能动手

## 本课目标

前两课你已经知道了入口和安全边界，这一课我们终于进入 Agent 最有辨识度的一部分：工具系统。

大模型本身擅长生成文本，但它不能天然读取你本地的文件，也不能直接修改代码，更不能自动执行终端命令。Agent 之所以从“聊天机器人”升级为“可执行系统”，核心就在于工具。

学完这一课，你应该能说清楚：

- 工具在 Agent 体系里到底扮演什么角色。
- 当前项目为什么要把工具拆成文件、终端、代码搜索、Git 四组。
- 一个合格的 Agent 工具至少应该包含哪几部分。

## 一、先建立一个核心认知

**工具不是“给模型的外挂”，而是“给模型开放的能力接口”。**

模型本身不会直接碰你的文件系统和终端，它只能：

1. 看到工具说明。
2. 决定是否调用某个工具。
3. 按工具 schema 组织参数。
4. 读取工具返回结果。
5. 再决定下一步怎么做。

所以工具系统本质上是在做“能力协议设计”。

## 二、当前项目为什么分成四类工具

### 1. 文件工具 `file-tools.ts`

这是最基础的一组能力，因为代码助手几乎绕不开文件操作。当前提供了：

- `read_file`
- `write_file`
- `edit_file`
- `list_directory`

这组工具让 Agent 具备“理解上下文”和“落地修改”的基础能力。

### 2. 终端工具 `exec-tools.ts`

有些动作仅靠读写文件无法完成，比如：

- 安装依赖
- 运行测试
- 执行构建
- 调试命令

所以需要 `execute_command` 作为操作系统层面的桥梁。

### 3. 代码搜索工具 `code-tools.ts`

当代码库一大，逐个读文件会非常低效。`search_code` 用来快速定位某个模式在哪些文件里出现过，是一种低成本的代码导航能力。

### 4. Git 工具 `git-tools.ts`

一个能改代码的 Agent，如果完全看不到版本状态，其实很危险。Git 相关工具让它至少可以：

- 看当前变更
- 查看 diff
- 暂存文件

这意味着它开始具备“对变更负责”的基础意识。

## 三、工具是如何被定义出来的

当前项目使用 `@langchain/core/tools` 的 `tool()` 方法定义工具，并且通过 `zod` 来声明 schema。

这里有两个非常适合教学强调的点。

### 1. schema 很重要

schema 不只是参数校验，它还承担了“告诉模型该怎么调用我”的作用。

一个工具至少要说清楚：

- 我叫什么
- 我是干什么的
- 我需要哪些参数
- 每个参数是什么意思

如果 schema 模糊，模型就更容易误用工具。

### 2. description 也很重要

工具描述不是写给程序员看的注释，而是写给模型看的能力说明。它会影响模型何时选择这个工具，以及如何组织调用。

这也是为什么教程里要把“写工具”讲成一项设计工作，而不是单纯编码工作。

## 四、一个工具最少应包含四部分

这一点你需要记住，因为它几乎可以作为你以后设计工具的检查清单。

### 输入

通过 schema 定义清楚参数结构。

### 执行

真正完成文件读写、命令执行、搜索或 Git 操作。

### 校验

在执行前做安全检查，例如路径限制、命令白名单、文本存在性判断等。

### 输出

以一致、可被模型理解的方式返回结果。

当前项目里，绝大多数工具最后都返回字符串，这样实现简单，也方便模型直接消费。

## 五、逐组看当前工具设计的教学价值

### 文件工具：最适合教学入门

文件工具是最容易让学员理解的一组，因为行为直观。

比如 `read_file`：

- 先做路径校验
- 再拼接绝对路径
- 尝试从缓存读取
- 读取文件内容
- 更新缓存
- 返回字符串结果

这组流程很像一个完整的工具模板，特别适合第一批练习。

### 终端工具：最能体现风险与边界

`execute_command` 的实现提醒我们：工具不是“能做就行”，还要考虑：

- 命令是否合法
- 工作目录是否合法
- 是否设置超时
- 输出缓冲区是否有限制

这正是工程化 Agent 和演示型脚本的差别。

### 代码搜索工具：展示效率优化思路

`search_code` 没有逐个手工打开文件，而是先根据 glob 找到一批目标文件，再按文本模式扫描内容。

这体现出一个很重要的认知：**Agent 不一定总靠大模型推理，有时先做简单、高效的程序化处理，反而更好。**

### Git 工具：补上“变更感知”

Git 工具的意义不在于功能多，而在于它让 Agent 开始具备最基本的版本控制意识。即便现在只有 status、diff、add，也足以帮助学员理解“代码改动应该被检查，而不是盲改”。

## 六、为什么 createAgent 要先创建工具再创建模型 Agent

这个顺序不是随便写的。

因为在 `createReactAgent()` 之前，系统需要先明确一件事：**本次运行中，模型到底能使用哪些动作能力。**

也就是说，工具列表不是附加信息，而是 Agent 构造参数的一部分。它们共同定义了这个 Agent 的能力边界。

## 七、当前工具系统的局限

从教程角度，局限同样值得讲清楚。

### 1. 返回值基本是字符串

这实现简单，但不够结构化。未来如果要做更复杂的 UI 展示，可能需要返回更细的结构数据。

### 2. 工具还不支持更强的代码理解

当前 `search_code` 更接近文本搜索，还没有 AST 分析、符号级导航等高级能力。

### 3. 工具之间还没有形成复杂协作协议

现在主要是模型自己决定串联调用工具，工具本身没有额外共享上下文协议。

## 本课小结

工具系统是整个 Agent 项目从“能说”到“能做”的分水岭。

你应该记住三件事：

- 工具是能力协议，不只是函数封装。
- schema、description、校验、输出同等重要。
- 工具数量不是重点，重点是是否形成了清晰、可控、可扩展的能力边界。
