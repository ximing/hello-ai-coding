# 第 6 课：Prompt 与状态，决定 Agent 怎么想、记什么

## 本课目标

很多人提到 Agent，第一反应是模型和工具；再进阶一点，会开始讨论工作流。但真正让一个 Agent 逐渐稳定、可控、可扩展的，往往是另外两层：

- Prompt：告诉模型该如何行动
- State：告诉系统当前记住了什么

这两者经常被混在一起讲，但它们不是同一件事。本课的目的，就是把这条边界讲清楚。

学完这一课，你应该能清晰回答：

- Prompt 为什么是行为约束，而不是运行状态。
- State 为什么是系统侧的数据结构，而不是给模型看的说明书。
- 当前项目里，哪些状态已经接上主流程，哪些仍然是未来扩展位。

## 一、先分清 Prompt 和 State

这是本课最重要的一句话：

**Prompt 主要解决“怎么想、怎么做”的问题，State 主要解决“现在记住了什么”的问题。**

如果再通俗一点：

- Prompt 更像岗位说明书
- State 更像工作记录本

前者告诉 Agent 应该遵守什么规则，后者记录 Agent 在这次运行里已经积累了哪些信息。

## 二、当前项目里的 Prompt 在做什么

`src/agent/prompt.ts` 提供了 `createSystemPrompt()`，它会基于当前工作目录和可用工具列表，生成一段 system prompt。

这个 Prompt 主要覆盖了几类内容：

### 1. 角色定义

它首先告诉模型：你是一个 AI coding assistant CLI tool。

角色定义的作用是限制模型的身份视角。没有这层角色约束，模型回答很容易发散，甚至忽略工具系统的存在。

### 2. 工作流程约束

Prompt 里明确要求模型：

- 先理解需求
- 再做规划
- 然后合理使用工具
- 遵守最佳实践
- 注意安全边界
- 清晰沟通进展

这相当于给模型注入了一套“做事方法论”。

### 3. 工具感知

Prompt 明确列出了可用工具，例如：

- `read_file`
- `write_file`
- `edit_file`
- `list_directory`
- `execute_command`
- `search_code`
- `git_status`
- `git_diff`

这很关键，因为模型不是自动知道自己拥有哪些工具。它必须先被告知能力边界，才能在推理时考虑调用。

### 4. 安全提醒

Prompt 还会强调：

- 不要无确认删除文件
- 不要随意执行破坏性命令
- 路径操作要验证
- 要待在工作目录内

这里的意思不是“只靠 Prompt 做安全”，而是让模型在行为层也具备安全倾向。

## 三、为什么 Prompt 很重要但又不够

这是学习 Agent 时很容易误判的点。

Prompt 很重要，因为它影响：

- 回答风格
- 工具选择倾向
- 规划习惯
- 安全意识

但 Prompt 不够，因为它本质上仍然是自然语言约束。真正的系统安全和运行正确性，不能只靠模型“自觉遵守”，还需要程序侧的校验。

所以你应该形成一个成熟认知：

- Prompt 负责软约束
- 程序逻辑负责硬约束

这也是为什么上一课我们要讲安全模块，这一课要讲 Prompt。二者不是替代关系，而是互补关系。

## 四、当前项目里的状态设计在做什么

接着看 `src/agent/state.ts`。

这里定义了一个 `AgentState` 接口，并提供了 `createInitialState()`。从字段上看，它已经开始为更复杂的 Agent 运行过程做建模。

当前包含的典型字段有：

- `messages`
- `workingDirectory`
- `fileCache`
- `executionPlan`
- `taskStatus`
- `currentTool`
- `toolResults`

这说明设计者已经在考虑一个更完整的运行时状态系统。

## 五、这些状态字段分别意味着什么

### `messages`

代表消息历史。这通常是 Agent 对话系统最基础的状态。

### `workingDirectory`

代表当前运行上下文的工作范围。

### `fileCache`

表示某些文件内容可能被缓存，避免重复读取。

### `executionPlan`

这是很值得关注的字段。它意味着系统希望显式记录“执行计划”，而不仅仅是把计划留在模型脑子里。

### `taskStatus`

用于标识当前任务所处阶段，例如：

- planning
- executing
- completed
- error

### `currentTool`

表示当前正在执行的工具，有助于 UI 展示或调试追踪。

### `toolResults`

用于记录工具执行历史，这在审计、调试、可视化场景中都很有价值。

## 六、哪些状态已经真正生效，哪些还在预留

这部分非常重要，因为教程不能只讲“设计得多漂亮”，还要讲“现在到底用了多少”。

### 已经实际生效的部分

当前项目真正跑起来的主链路，主要依赖的是：

- LangGraph 自身的消息和 checkpoint 机制
- `AgentRunner` 持有的 `sessionId`、`workingDirectory`、`fileCache`
- UI 层的本地 `useState()` 状态

### 尚未完整接入的部分

`src/agent/state.ts` 里定义的这套 `AgentState`，目前更像是“未来系统状态模型”，而不是已经完整接管运行主流程的唯一状态源。

换句话说：

- 它不是无用代码
- 但它也还不是当前系统真正的状态中枢

这正是一个典型的极简版项目特征：设计方向已经出现，但实现还在渐进接入中。

## 七、为什么这一课对教程很重要

因为很多初学者会把 Agent 误解为“模型 + 工具”二元结构。事实上，一旦项目开始变复杂，就一定会遇到：

- 模型行为不稳定，Prompt 要不要拆层？
- 运行过程变长，状态要不要显式建模？
- 工具调用多了，历史结果怎么记录？
- UI 想展示计划和进度，数据从哪里来？

所以 Prompt 和 State 不是高级可选项，而是系统从 demo 向产品演进时的必经阶段。

## 八、从课程设计角度，为什么要把两者放在同一课

因为它们刚好形成一组对照：

- Prompt 偏模型侧
- State 偏系统侧
- Prompt 偏行为策略
- State 偏运行数据
- Prompt 偏自然语言
- State 偏结构化对象

把两者放在一起学，反而更容易建立边界感。

## 本课小结

这一课最重要的结论是：

- Prompt 决定 Agent 倾向于怎样行动。
- State 决定系统能记住哪些运行信息。
- 当前项目两者都已经出现，但成熟度不同。

Prompt 已经真实影响运行行为，而 `AgentState` 更多还处于预留与逐步接入阶段。

下一课我们会进入 UI 层，看这些行为和状态信息是如何被展示给用户的。因为一个真正可用的 Agent，不应该只是“在内部运行”，还应该把过程清楚地暴露出来。
