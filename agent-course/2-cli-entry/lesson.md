# 第 2 课：从 CLI 入口搭起 Agent 外壳

## 本课目标

上一课我们先建立了项目全局地图。这一课开始真正进入代码，从 `src/cli.ts` 出发，看一个命令行 Agent 是如何被启动起来的。

很多人学 Agent 时，注意力只放在模型和 Prompt 上，但真实项目里，最外层的 CLI 外壳同样重要。因为用户的任务、配置、模式切换、调试参数，都是从这里进入系统的。

学完这一课，你应该能清楚说明：

- 命令行参数是如何被解析的。
- 为什么要先加载配置再创建 Agent。
- 为什么 `render(CliApp)` 是交互模式的真正起点。

## 先理解 CLI 的职责

`src/cli.ts` 不是“Agent 本体”，而是“Agent 启动器”。

它主要做四类事：

1. 解析输入：识别用户到底传了什么任务和选项。
2. 准备环境：加载 `.env`、读取配置、确定工作目录。
3. 创建运行对象：实例化 Agent 和 AgentRunner。
4. 启动交互界面：把 Agent 交给 Ink UI 去驱动。

你可以把它理解成一个应用层 orchestrator。它本身不负责推理，但负责把推理系统拉起来。

## Commander 在这里扮演什么角色

项目使用 `commander` 来定义命令结构。它的优势是很适合把一个 CLI 工具拆成：

- 默认命令
- 子命令
- 参数选项
- 帮助信息

在当前项目里，默认命令既可以接受一个单次任务，也可以进入交互式会话。这是一个很典型的 Agent CLI 设计，因为它兼容了两种使用方式：

- 一次性执行：适合快速提问
- 持续对话：适合探索式开发

## 默认命令的启动链路

先看默认命令的心智模型：

### 第一步：收集参数

程序定义了这些关键参数：

- `task`：单次执行任务描述
- `--model`：指定模型
- `--working-dir`：指定工作目录
- `--verbose`：打开更多日志
- `--dry-run`：只预览配置，不真正执行
- `--config`：指定配置文件路径

这些参数决定了本次运行的行为边界。

### 第二步：生成配置覆盖项

CLI 不会直接把命令行参数散着传给底层，而是先整理成 `configOverrides`。这样做的好处是：

- 统一输入来源
- 和默认配置、用户配置、环境变量配置一起合并
- 让底层只面对一个最终配置对象

这是典型的工程化思路：**不要让业务逻辑关心“这个值到底来自命令行还是环境变量”，而是提前在入口层完成归一化。**

### 第三步：加载完整配置

`loadConfig()` 会把多种来源合并成最终运行配置。CLI 层只负责把局部覆盖项传进去。

这一层的好处是，你未来想支持更多配置源时，不需要改 Agent 主体逻辑，只需要在配置系统里扩展即可。

### 第四步：创建 Agent 和 Runner

入口文件里调用了 `createAgent()`，拿到：

- `agent`
- `sessionId`
- `workingDirectory`
- `fileCache`

随后又创建了 `AgentRunner`。这里要特别注意：

- `agent` 更像底层执行引擎
- `AgentRunner` 更像适配外部调用的包装器

这个包装器把 `invoke()` 和 `stream()` 这样的调用方式整理成更适合 UI 层使用的接口。

### 第五步：决定是否真正运行

如果开启了 `--dry-run`，程序会打印配置并退出，而不是进入真正的 Agent 执行流程。

这是一个很好的教学点，因为它体现了真实工具中的“预览模式”思想：先确认运行环境，再做真正执行。

### 第六步：渲染 Ink UI

如果不是 dry-run，程序会调用 `render()` 挂载 `CliApp`。

这一刻，应用才真正进入交互式运行状态。后续的输入监听、消息显示、工具状态展示，都由 UI 层接管。

## 为什么入口层要这么设计

### 1. 让外部输入先被标准化

命令行参数、配置文件、环境变量，是三种完全不同的输入方式。如果不在 CLI 层统一收口，后面代码会越来越乱。

### 2. 让 Agent 创建逻辑保持纯粹

`createAgent()` 负责创建 Agent，不需要顺便处理大量命令行细节。这叫职责分离。

### 3. 让 UI 与启动逻辑解耦

CLI 负责准备运行环境，UI 负责交互展示。二者分开之后，未来你要做 Web UI、TUI、甚至 API Server 都更容易。

## 当前 CLI 里有哪些值得关注的“半成品”

这也是教程里很值得讲的地方。当前项目不是一个完全成熟的 CLI 产品，因此你能看到几个明显的演进位：

### `continue`

这个命令已经有接口定义，但目前只是打印说明，真正的会话恢复还没做完。

### `list-sessions`

这个命令也还只是占位，说明项目已经意识到“会话管理”是重要能力，但还没有进入完整实现阶段。

### `clean`

同样属于保留接口。它提醒我们：一个真实 Agent 工具不仅要能跑，还要有清理缓存、管理状态的能力。

### `init`

这个命令则更完整一些，它会生成一个示例配置文件。这非常适合放进教程，因为它能让学员理解“一个工具如何自举配置”。

## 你应该形成的三个判断

学完这一课，请务必形成下面三个工程判断：

### 判断 1：CLI 不等于壳子，它是系统的输入边界

只要一个系统有输入边界，就值得被认真设计。Agent 项目也不例外。

### 判断 2：配置归一化应该发生在最外层

不要把“从哪里来的值”这个问题，带进系统内部。

### 判断 3：交互式 UI 是在启动之后才接管控制流

也就是说，先有程序初始化，后有交互式会话，而不是反过来。

## 本课小结

这一课你要建立的核心认知是：`src/cli.ts` 不是一个零散的命令集合，而是整个 Agent 系统的启动编排层。

它决定：

- 用户如何把需求交进来
- 配置如何被整理
- Agent 何时被创建
- UI 何时开始接管

接下来我们会继续向内走，下一课看配置系统和安全边界。因为一旦 Agent 有“动手能力”，你就必须开始思考：它应该被允许做什么，不应该做什么。
