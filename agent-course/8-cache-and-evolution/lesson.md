# 第 8 课：缓存、局限与下一步演进

## 本课目标

前面七课，我们已经沿着这个极简 AI Agent 项目的主线，把入口、配置、安全、工具、装配、Prompt、状态、UI 都拆了一遍。最后一课要做两件事：

1. 回看当前系统已经具备哪些“像产品”的能力。
2. 明确它距离更完整的 Agent 教程项目还差什么。

这也是教程里非常重要的一步。因为学习不应该停留在“看懂已有代码”，还应该升级到“知道下一步怎么做”。

## 一、为什么最后一课要讲缓存

缓存看起来不像 Agent 的主角，但它非常适合拿来做“从 demo 走向实用”的切入点。

因为当系统开始频繁读文件、重复访问上下文时，如果每次都重新读取、重新计算，性能和体验都会明显下降。尤其对 Coding Agent 来说，文件读取是高频动作，缓存能直接减少重复成本。

所以 `src/utils/file-cache.ts` 虽然只是一个辅助模块，却代表着一个重要信号：这个项目已经不再只追求“能跑”，而开始考虑“是否高效”。

## 二、当前缓存系统做了什么

`FileCache` 的设计非常适合教学，因为它简单、清楚，又足够体现真实工程思维。

它主要做了几件事：

### 1. 用 `Map` 存储缓存条目

每个条目记录：

- 文件内容 `content`
- 修改时间 `mtime`
- 文件大小 `size`

这说明缓存不是只存文本，而是同时保留了用于失效判断和统计的信息。

### 2. 在读取时检查文件是否变化

`get()` 并不是简单从 `Map` 里拿值，而是会：

- 先看缓存里有没有
- 再读取文件状态
- 比较 `mtime`
- 如果文件已经变了，就让缓存失效

这体现出一个核心原则：**缓存不能只追求快，还必须尽量保证正确。**

### 3. 维护命中和未命中统计

系统会记录：

- `hits`
- `misses`
- `hitRate`

这让缓存不再是一个黑盒优化，而是一个可被观察、可被衡量的性能机制。

### 4. 支持失效与清空

通过 `invalidate()` 和 `clear()`，系统为后续演进预留了很好的接口。

## 三、缓存为什么能作为“工程化程度”的标志

因为缓存背后体现的是一种思维升级：

- demo 阶段只关心“第一次能不能跑通”
- 工程阶段开始关心“重复运行是否稳定高效”

对 Agent 来说，这种升级很重要。因为 Agent 常常会：

- 多次读取同一个文件
- 在多轮会话里反复访问上下文
- 执行链条较长，重复成本会被放大

所以哪怕只是一个简单文件缓存，也已经说明这个项目开始向“更像真实工具”迈进。

## 四、回看整个项目，当前已经解决了什么

现在适合做一次全局回顾。当前极简版已经具备了一个教程项目最重要的骨架：

### 1. 有完整主链路

从 CLI 输入任务，到 Agent 装配、工具调用、UI 展示，已经形成了最小闭环。

### 2. 有基础工程分层

入口、Agent、工具、UI、配置、安全、缓存，各自职责比较清晰，适合教学。

### 3. 有初步安全意识

命令白名单、路径限制、危险模式拦截，都已经存在。

### 4. 有流式交互体验

UI 能显示消息、当前动作、工具结果，用户能够看到执行过程。

### 5. 有未来演进接口

会话恢复、状态扩展、确认机制、更多工具、技术方案文档，这些都已经露出了方向。

## 五、当前还明显缺什么

这是本课最重要的现实判断。

### 1. 会话恢复仍不完整

虽然有 `sessionId` 和 `MemorySaver`，也有 `continue` 命令，但真正的跨运行恢复和会话管理还没有闭环。

### 2. 风险确认机制还未真正打通

系统知道某些操作“应该确认”，但还没有完整的人机确认流和审批体验。

### 3. 状态模型还没有成为单一事实源

`AgentState` 已经设计出来，但仍未完整贯穿执行主链路。

### 4. 工具能力仍偏基础

当前工具足够教学，但距离更强的工程辅助还有空间，例如：

- 更精细的代码搜索
- AST 级分析
- 测试运行与结果解析
- 更安全的命令执行策略

### 5. 测试与评估体系不足

一个教程项目可以先不把测试做到很重，但如果要进一步演进，单元测试、集成测试、Agent 行为评估都会变得重要。

## 六、为什么这些缺口反而很适合做教程进阶课

因为一个好的教程，不是把所有东西一次性堆满，而是保留明确的生长路径。

当前项目最大的优势之一，就是它已经有了可运行的最小系统，同时又保留了很多非常自然的下一步。

这使得你很容易设计“基础课 -> 进阶课”的教学路线。

## 七、如果继续扩展，我建议的优先级

如果你准备把这个教程继续做成更完整的系列，我建议按下面顺序演进。

### 优先级 1：补齐会话持久化

原因：

- 它和当前已有 `sessionId`、`MemorySaver`、`continue` 命令天然衔接
- 会让系统从“一次性工具”更像“持续协作助手”
- 对学员理解 Agent 的长期上下文非常有帮助

### 优先级 2：补齐确认与审批流

原因：

- 这是 Agent 真正进入执行层之后最关键的安全补丁
- 非常适合讲“模型意图”和“系统许可”之间的边界

### 优先级 3：让状态模型贯穿全链路

原因：

- 这能把执行计划、当前工具、工具历史、任务阶段更清楚地暴露出来
- 对 UI 展示和调试都很有帮助

### 优先级 4：增强工具系统

例如补充：

- 更强的代码搜索
- 文件 diff 工具
- 测试运行解析
- 项目结构分析工具

### 优先级 5：增加测试与评估

当功能越来越多时，测试和评估不再是“锦上添花”，而是保证教程代码可持续演化的基础。

## 八、如何把这一版教程升级成下一版课程体系

你完全可以把当前 8 课当成基础篇，然后追加一个进阶篇，例如：

- 进阶 1：会话持久化与恢复
- 进阶 2：危险操作确认与人机协作
- 进阶 3：状态驱动的可观察 Agent
- 进阶 4：更强的搜索与代码理解能力
- 进阶 5：为 Agent 增加测试与评估框架

这样课程结构会非常自然：

- 基础篇解决“看懂并搭起最小 Agent”
- 进阶篇解决“把最小 Agent 做成更完整系统”

## 整个系列回顾

如果你已经学完这 8 课，那么你应该已经能从整体上理解一个极简 AI Coding Agent 的构成：

- 如何从 CLI 接收任务
- 如何加载配置并设定安全边界
- 如何通过工具获得执行能力
- 如何把模型、工具、Prompt 和会话装配成 Agent

- 如何识别当前系统的边界与下一步演进方向

## 本课小结

最后一课最重要的收获，不是缓存本身，而是“系统视角”。

你不再只是看某一个文件做了什么，而是开始判断：

- 这个系统已经成熟到什么程度
- 哪些能力是基础骨架
- 哪些能力仍在预留阶段
- 下一步怎样扩展最合理

当你具备这种视角时，你就不只是在学一个项目，而是在学如何设计和演进一个真正的 AI Agent 系统。
