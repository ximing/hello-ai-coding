# 第 5 课：把 LLM、工具和会话真正组装成 Agent

## 本课目标

前面几课你已经分别看过入口、配置、安全和工具。这一课开始进入整个项目最核心的地方：`src/agent/agent.ts`。

如果说前面的内容是在认识零件，那么这一课就是在看装配过程。因为真正让系统从“一堆模块”变成“一个可运行 Agent”的，不是某一个单独文件，而是它们被组装起来的方式。

学完这一课，你应该能回答：

- `createAgent()` 到底做了哪些关键步骤。
- 为什么工具、模型、Prompt、会话能力要在同一个阶段完成装配。
- `AgentRunner` 为什么是一个有必要存在的包装层。

## 一、先建立一个装配视角

阅读 `src/agent/agent.ts` 时，不要把它看成“写法很多的实现文件”，而要把它看成“系统集成中心”。

这个文件的真正任务不是发明新能力，而是把前面已经存在的能力拼接成一个运行时对象。

你可以把它理解成这样一条链：

`配置 -> 安全与缓存 -> 工具 -> 模型 -> Prompt -> Agent -> Runner`

这条链的顺序本身就很有教学价值。

## 二、`createAgent()` 的执行顺序

### 第一步：确定工作目录

函数一开始会优先使用 `agentConfig.workingDirectory`，否则回退到 `appConfig.workingDirectory`。

这一步决定了当前 Agent 的活动边界。后面很多工具的安全验证、路径解析、命令执行都会依赖这个目录。

### 第二步：初始化基础组件

这里创建了两个重要对象：

- `SecurityValidator`
- `FileCache`

前者负责能力边界，后者负责性能优化。它们都不是 Agent 的“主角”，但没有它们，Agent 的运行质量会明显下降。

### 第三步：创建工具集合

接着系统分别创建：

- 文件工具
- 终端工具
- 代码工具
- Git 工具

注意这里不是简单 import，而是把 `workingDirectory`、`security`、`fileCache`、`appConfig` 等上下文依赖一起注入进去。

这说明工具不是静态函数，而是“绑定了当前运行环境”的能力对象。

### 第四步：组装统一工具数组

随后，这些具体工具被整理成一个 `tools` 数组。这个数组非常关键，因为它直接决定了后面创建的 Agent 拥有哪些外部动作能力。

换句话说：**工具数组就是当前 Agent 的可执行能力清单。**

### 第五步：初始化 LLM

接下来系统通过 `initChatModel()` 初始化聊天模型，并根据配置动态拼出：

`provider:model`

例如：

- `anthropic:claude-3-5-sonnet-20241022`
- `openai:gpt-4.x`

同时还会根据配置补充：

- `temperature`
- `maxTokens`
- `apiKey`
- `baseURL`

这意味着模型层是可替换的，而不是和项目强耦合的。

### 第六步：创建 system prompt

在模型和工具都准备好之后，系统调用 `createSystemPrompt()`，把：

- 当前工作目录
- 可用工具名列表

传进去，生成最终的 system prompt。

这一步背后的思想很重要：Prompt 不是孤立文本，而是要和当前运行环境绑定。尤其是工具列表，如果 Prompt 不知道当前有哪些工具，就无法引导模型正确调用能力。

### 第七步：初始化 checkpointer

当前项目使用 `MemorySaver()` 作为 checkpoint saver。它提供了基本的会话持久能力，但仍然主要是内存级别的轻量实现。

这说明系统已经开始考虑“多轮会话”问题，而不是每次都做完全无状态的一次性调用。

### 第八步：创建 ReAct Agent

最后，使用 `createReactAgent()` 完成真正的 Agent 实例化。

这里会把下面几个东西统一传入：

- `llm`
- `tools`
- `checkpointSaver`
- `messageModifier`

到这一步，一个能进行工具调用的 ReAct Agent 才真正诞生。

## 三、为什么这里选择 ReAct Agent

当前项目选择的是 LangGraph 预构建的 ReAct Agent，这非常适合极简版教程。

原因有三点：

### 1. 结构清晰

ReAct 的基本模式就是：

- 思考
- 调用工具
- 读取结果
- 继续思考

这与“教学中的逐步推导”非常一致。

### 2. 足够实用

对一个极简版 Coding Agent 来说，ReAct 已经能支撑大量基础场景，例如：

- 先读文件
- 再搜索代码
- 再执行命令
- 最后汇总结论

### 3. 扩展空间大

虽然当前用的是预构建 Agent，但未来完全可以继续演进为更复杂的图结构、多节点编排或多 Agent 协作。

## 四、`sessionId`、`thread_id` 和会话概念

这一课很适合讲“会话”的三个层次。

### `sessionId`

这是当前应用层给每个会话生成的标识。它主要用于区分不同运行会话。

### `getThreadConfig()`

这个函数会把 `sessionId` 包装成 LangGraph 需要的 `thread_id` 配置。

### `MemorySaver`

它负责在这个 thread 上保存对话检查点，从而支持多轮上下文延续。

你可以这样理解：

- `sessionId` 是应用层命名
- `thread_id` 是框架层会话键
- `MemorySaver` 是会话存储实现

这三者组合起来，才形成当前项目的轻量会话机制。

## 五、为什么还需要 `AgentRunner`

很多初学者看到这里会问：既然已经有 `agent` 了，为什么还要再包一层 `AgentRunner`？

这是个非常好的问题。

`AgentRunner` 的价值主要有三点：

### 1. 屏蔽底层调用细节

UI 层并不需要知道 thread config、invoke 参数结构、streamMode 这些底层框架细节。它只需要：

- `invoke(input)`
- `stream(input)`
- `getSessionInfo()`

### 2. 聚合会话信息

`AgentRunner` 持有：

- `sessionId`
- `workingDirectory`
- `fileCache`

因此它能为上层 UI 提供更完整的运行信息，而不只是“帮你调一下 agent”。

### 3. 形成更稳定的外部接口

将来如果底层 Agent 实现更换了，只要 `AgentRunner` 的外部接口稳定，UI 层就不一定需要大改。

这就是典型的包装器价值：隔离变化。

## 六、当前装配逻辑里隐藏的设计思想

### 1. 依赖注入

工具不是自己去全局取配置，而是在创建阶段被注入依赖。这样更清晰，也更方便测试。

### 2. 能力边界显式化

通过 `tools` 数组，Agent 有哪些能力是一眼可见的，而不是分散在代码各处。

### 3. 框架与业务的适配分层

LangGraph 是底层框架，`createAgent()` 则是本项目对框架的本地化适配层。

## 七、这一层仍然有哪些演进空间

虽然当前 `agent.ts` 已经很适合做极简版教程，但它仍然有不少可继续扩展的地方：

- 更强的状态管理接入
- 更真实的会话恢复
- 更细的工具调用日志
- 更可配置的 Prompt 模板机制
- 更复杂的工作流图，而不只是单一 ReAct 模式

这也说明：这一课不是“看完装配就结束”，而是为后续演进打基础。

## 本课小结

这一课最重要的结论是：Agent 并不是一个单独“生成出来”的对象，而是被一步步装配出来的。

在当前项目里，`src/agent/agent.ts` 就承担了这个装配中心角色。它把：

- 配置
- 安全
- 缓存
- 工具
- 模型
- Prompt
- 会话

统一整合成一个真正可运行的系统。

下一课我们会继续深入，专门讲 Prompt 与状态，因为这两者决定了：Agent 会如何思考，以及系统会如何记录运行中的信息。
