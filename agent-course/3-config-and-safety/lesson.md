# 第 3 课：配置系统与安全边界

## 为什么这一课很关键

从这一课开始，你会真正意识到：Agent 项目不是“把模型接进来”就结束了。只要一个系统开始读文件、执行命令、修改代码，它就必须面对两个现实问题：

- 它应该如何被配置。
- 它应该被允许做什么。

前者关系到系统灵活性，后者关系到系统安全性。真实工程里，这两部分往往决定一个 Agent 能不能从 demo 变成可信工具。

## 本课目标

学完这一课，你应该能清楚说明：

- 默认配置、用户配置、环境变量、命令行覆盖之间的关系。
- 为什么命令白名单和路径限制是最基础的安全边界。
- 为什么当前项目已经有安全意识，但还没有实现完整的审批闭环。

## 一、配置系统解决的是什么问题

如果没有配置系统，每次运行 Agent 都要把模型、工作目录、命令白名单、UI 选项硬编码在程序里。这会导致三个问题：

1. 用户无法按场景切换配置。
2. 环境变化时必须改代码。
3. 工具难以迁移和复用。

所以 `src/utils/config.ts` 的任务，就是把“运行时可变项”从代码中抽离出来，形成一个统一的配置入口。

## 二、先看默认配置

`defaultConfig` 里定义了当前项目的基础运行方式，主要包含四大块：

### 1. LLM 配置

这里定义了：

- provider
- model
- temperature
- maxTokens

这说明模型不是写死的，而是可切换的。

### 2. workingDirectory

这个字段决定 Agent 的主要工作范围。很多文件操作和命令执行，都会围绕这个目录展开。

### 3. security

这是本课的重点。当前默认安全配置包含：

- `allowedCommands`
- `restrictedPaths`
- `requireConfirmation`

你可以把它们理解成三层边界：

- 哪些命令可以执行
- 哪些路径不允许访问
- 哪些危险操作即使可识别，也应该进一步确认

### 4. tools 与 ui

前者控制工具层的参数，例如文件大小限制、忽略目录、终端超时；后者控制界面行为，例如是否显示思考过程、是否进入 verbose 模式。

## 三、配置是如何合并的

项目里最值得学习的地方之一，是配置并不是单一来源，而是多来源合并。

当前大致有四层：

1. 默认配置 `defaultConfig`
2. 用户配置文件 `loadUserConfig()`
3. 环境变量配置 `loadEnvConfig()`
4. CLI 传入的覆盖项 `overrides`

最终由 `mergeConfig()` 合并成一份 `AppConfig`。

## 四、为什么配置要分层覆盖

这种设计不是“写法好看”，而是为了解决真实使用问题。

### 默认配置

保证工具开箱可用。

### 用户配置文件

保证用户可以保存长期偏好，而不用每次都手动输入。

### 环境变量

适合存放 API Key、Base URL 这类不应直接写进源码的值。

### CLI 覆盖项

适合本次运行的临时调整，例如临时换模型、临时改工作目录。

这四层叠起来，工具才既灵活又可控。

## 五、安全边界是怎么实现的

现在进入 `src/utils/security.ts`。这个文件实现的是一套非常朴素但很重要的防线。

### 1. 命令白名单

`validateCommand()` 会先取出命令的第一个词，也就是主命令，然后检查它是否在允许列表里。

这意味着：

- `npm test` 可能允许
- `python script.py` 如果不在白名单，就不允许
- `rm -rf ...` 即使写出来，也会先在白名单层被挡住

这是一种非常基础但非常有效的做法。因为 Agent 一旦可以执行任意 shell 命令，风险会立即放大。

### 2. 危险模式匹配

当前实现还额外拦截了一些明显危险的命令模式，例如：

- 删除根目录
- fork bomb
- 磁盘格式化
- 原始磁盘写入

这说明安全不是只靠白名单，还要针对少数极高风险模式做额外兜底。

### 3. 路径校验

`validatePath()` 会把用户传入的路径解析成绝对路径，然后做两类检查：

- 是否还在工作目录内
- 是否命中了受限系统路径

这是 Agent 文件操作的关键安全边界。

因为只要允许它随意越出工作目录，后果就不再局限于当前项目，而可能影响整台机器上的其他文件。

## 六、当前安全体系的边界在哪里

这一点特别适合放进教程。因为真实项目里，知道“已经做了什么”固然重要，知道“还没做什么”同样重要。

### 已经有的部分

- 命令白名单
- 路径限制
- 危险模式识别
- 文件大小限制接口
- 风险操作确认配置项

### 还缺的部分

- 真正的人机确认交互流
- 更细粒度的命令参数审计
- 不同工具级别的权限策略
- 审计日志与安全事件记录

特别是 `requireConfirmation`，目前更多还停留在配置层和工具层认知中，尚未变成一个完整的运行时审批流程。

这也正是为什么当前项目适合做教程：它已经有了正确的方向，但保留了很多值得继续演进的空间。

## 七、从教学角度怎么讲这部分

很多教程会把“安全”讲成补充话题，但对 Agent 来说，它应该是主线内容。原因很简单：

- 普通聊天机器人输出文本，风险有限。
- Coding Agent 会读文件、改文件、跑命令，风险直接进入执行层。

所以在教学时，建议把安全边界作为“Agent 能力的一部分”来讲，而不是“上线前再补”。

## 本课小结

这一课你要带走两个重要认知：

第一，配置系统不是附属品，而是让 Agent 在不同环境中稳定运行的基础设施。

第二，安全边界不是可选优化，而是 Agent 获得执行能力之后必须同时引入的约束。

下一课我们就进入工具系统，看一个 Agent 是如何通过工具，从“会回答”升级为“会动手”的。
