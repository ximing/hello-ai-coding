# 第 7 课：流式 UI，把 Agent 的过程展示出来

## 本课目标

到目前为止，我们已经知道：Agent 有入口、有配置、有工具、有 Prompt，也有一定的会话和状态概念。但如果这些都只发生在内部，用户感知到的系统仍然像黑盒。

本课要讲的是：为什么 UI 在 Agent 项目中不仅仅是“显示结果”，而是“把执行过程可视化”的关键层。

学完这一课，你应该能清楚说明：

- Ink 在这个项目里承担了什么角色。
- `CliApp` 如何处理输入、启动执行、消费流式结果。
- 为什么“展示当前动作和工具结果”会显著提升 Agent 的可理解性。

## 一、为什么 Agent 特别需要过程型 UI

普通 CLI 工具很多时候只需要输出最终结果，例如一个 `ls` 或 `git status`。但 Agent 不一样，它经常包含：

- 多轮推理
- 多次工具调用
- 中间状态变化
- 非瞬时执行过程

如果只输出最后一句答案，用户会很难判断：

- 它到底有没有认真看代码
- 它是不是调用了工具
- 工具执行到了哪一步
- 为什么这次响应比较慢

所以 Agent 的 UI 价值之一，就是把“黑盒推理”尽可能变成“可观察执行”。

## 二、Ink 在这里解决什么问题

项目使用的是 `ink`，可以把 React 的组件式开发方式带到终端里。

这意味着你可以用熟悉的前端思维去做终端交互界面，例如：

- `useState()` 管理消息和输入状态
- `useEffect()` 处理副作用
- `useInput()` 监听键盘输入
- 组件拆分消息、会话信息、状态展示区域

这对教学很友好，因为很多人已经有 React 基础，用 Ink 会更容易理解终端 UI 的状态流动。

## 三、`CliApp` 的职责到底是什么

`src/ui/cli.tsx` 是当前交互层的核心组件。它大致负责四件事：

1. 管理用户输入
2. 管理消息历史
3. 启动 Agent 流式执行
4. 把执行状态映射到界面展示

你可以把它理解成“终端里的聊天页面 + Agent 运行控制器”。

## 四、先看输入处理

当前组件使用 `useInput()` 监听键盘输入，并处理一些典型行为：

- `Ctrl + C`：退出程序
- `Enter`：提交任务
- `Backspace/Delete`：删除输入
- 普通字符：继续拼接到输入框

这里的关键教学点是：UI 不是被动显示，它同时负责把用户行为转成系统事件。

## 五、`handleSubmit()` 是整节课的重点

当用户按下回车后，会进入 `handleSubmit()`。这一段逻辑非常值得逐行理解，因为它串起了 UI 与 Agent 执行链路。

### 第一步：读取并清洗输入

组件会先对输入做 `trim()`，空内容不提交。

### 第二步：处理本地命令

当前支持几个简单命令：

- `help`
- `clear`
- `status`
- `exit` / `quit`

这很像“聊天命令”和“系统命令”的区分。也就是说，不是所有输入都要交给 Agent，有些操作应该在 UI 本地直接处理。

### 第三步：写入用户消息历史

如果输入是正常任务，UI 会先把用户消息加到本地 `messages` 状态中。这样做的好处是，用户一提交就能看到自己的输入已经进入会话流。

### 第四步：切换处理状态

组件会设置：

- `isProcessing = true`
- 重置 `agentState`
- 清空输入框

这一步本质上是告诉界面：新的 Agent 执行已经开始。

### 第五步：消费 `agent.stream()`

这是最核心的一步。UI 并不是等 Agent 全部跑完后再一次性拿结果，而是通过异步迭代流式接收 chunk。

这意味着：

- 中间工具调用可以被实时感知
- 当前动作可以及时显示
- 工具结果可以边跑边展示
- 最终消息在结束后统一落进历史记录

这正是“流式 Agent UI”的核心体验差异。

## 六、流式 chunk 在当前实现里如何被使用

当前代码里，chunk 主要分成两类值得关注的信息：

### 1. `chunk.agent`

这部分通常携带 Agent 侧的消息。如果里面包含 `tool_calls`，UI 就会更新 `currentAction`，告诉用户当前正在执行哪个工具。

这一步非常关键，因为它把“模型决定调用工具”这个内部事件，转成了用户可见的反馈。

### 2. `chunk.tools`

这部分主要是工具执行结果。当前实现会把工具结果截断后放进 `results` 数组中展示。

虽然展示还比较简化，但它已经足够说明一个重要原则：**Agent 的工具行为应该尽量可见，而不是藏起来。**

## 七、为什么最终回答要晚一点写入消息历史

当前实现会在流式循环结束后，再把完整回答作为 assistant 消息追加到历史中。

这样设计的好处是：

- 用户不会看到一堆零散回答片段污染消息区
- 中间过程和最终答案的职责分离
- UI 结构更清楚：过程看状态区，结论看消息区

这是一个很值得学习的 UI 分层思路。

## 八、`components.tsx` 的拆分有什么价值

`src/ui/components.tsx` 把展示层拆成了多个小组件，例如：

- `SessionInfo`
- `Message`
- `AgentOutput`
- `ToolExecution`
- `Progress`
- `HelpInfo`

这种拆分不是为了“文件更长”，而是为了让不同展示职责彼此独立。

例如：

- 消息组件专注于谁说了什么
- 状态组件专注于执行进度和结果
- 帮助组件专注于命令说明

这也让未来 UI 迭代更容易，例如单独增强工具结果展示，而不影响消息结构。

## 九、当前 UI 做得不错的地方

### 1. 已经具备会话感

用户输入、系统消息、助手回答被统一组织成消息流，基本具备“聊天式 Agent”体验。

### 2. 已经具备执行感

工具调用名、结果摘要、完成状态，都会被展示出来。

### 3. 已经具备命令感

`help`、`clear`、`status`、`quit` 这些命令让它不像单纯问答框，而像一个真实 CLI 应用。

## 十、当前 UI 仍然可以怎么继续增强

### 1. 工具结果展示太短

现在只截取前 100 个字符，适合轻量提示，但不利于深入调试。

### 2. thinking 状态还不明显

虽然有 `thinking` 字段，但当前展示并不充分。

### 3. 缺少步骤级进度

如果一次任务涉及多个工具调用，用户目前还不容易形成完整步骤感。

### 4. 错误展示还可以更结构化

当前错误主要是文本显示，未来可以增加更清晰的分组和恢复建议。

## 本课小结

这一课最重要的结论是：UI 在 Agent 系统里不仅负责“显示内容”，更负责“让执行过程可观察”。

对用户来说，可观察性会直接影响信任感；对教程来说，可观察性会直接影响可理解性。

下一课我们会收束整个系列，讲缓存、局限和演进路线。也就是回答最后一个关键问题：这个极简版已经解决了什么，又还差什么。
