# 第 1 课：先看懂这个极简 AI Agent 在做什么

## 这一课要解决什么问题

很多人第一次看 AI Agent 项目时，会立刻钻进模型调用、Prompt、工具定义这些细节里，结果越看越碎。本课先不追求把每一行代码都看懂，而是先建立一张全局地图：这个项目从用户输入一条任务开始，到最后把结果显示在终端里，中间到底发生了什么。

学完这一课，你应该能回答三个问题：

1. 这个项目的最小闭环是什么。
2. 哪些模块是主干，哪些模块是增强项。
3. 为什么它已经是一个可运行的 Agent，但仍然只是“极简版”。

## 先把项目看成一条流水线

可以先把当前项目理解成一条非常清晰的流水线：

`用户输入任务 -> CLI 入口接收参数 -> 创建 Agent -> Agent 调用模型与工具 -> UI 流式展示结果`

对应到当前代码，大致可以映射成下面几层：

- `src/cli.ts`：程序入口。负责加载配置、创建 Agent、启动终端 UI。
- `src/agent/agent.ts`：装配中心。把 LLM、工具、会话能力拼成一个真正可用的 Agent。
- `src/tools/*.ts`：能力层。让 Agent 不只是“会说”，还可以“读文件、执行命令、搜索代码、查看 Git”。
- `src/ui/cli.tsx` 与 `src/ui/components.tsx`：展示层。把消息、工具执行状态、结果渲染到终端里。
- `src/utils/*.ts`：支撑层。提供配置加载、安全校验、文件缓存等基础能力。

如果你只记住一句话，那就是：**这个项目的本质不是“调用一次模型”，而是“把模型、工具、状态和界面组装成一个可交互的软件系统”。**

## 从用户视角走一遍主流程

### 第一步：用户发起任务

用户通过命令行执行 `ai-code "帮我分析这个项目"`，或者直接启动交互模式，然后在终端里输入任务。

此时，最先接住这条任务的是 `src/cli.ts`。这一层做的不是 AI 推理，而是应用启动工作：

- 读取 `.env`
- 解析命令行参数
- 加载配置
- 调用 `createAgent()`
- 创建 `AgentRunner`
- 用 Ink 渲染 `CliApp`

也就是说，CLI 是入口，但不是思考者。

### 第二步：系统把零件装成 Agent

真正把“模型 + 工具 + 会话”组装起来的是 `src/agent/agent.ts`。

这里做了几件关键事情：

- 根据配置确定工作目录
- 创建安全校验器 `SecurityValidator`
- 创建文件缓存 `FileCache`
- 创建四类工具：文件、终端、代码搜索、Git
- 初始化聊天模型 `initChatModel()`
- 生成 system prompt
- 使用 `createReactAgent()` 创建 ReAct Agent

这一步很重要，因为它定义了当前 Agent 的能力边界。模型并不是“天然什么都会”，它能执行哪些动作，完全取决于这里注册了哪些工具。

### 第三步：Agent 在运行时做推理与调用工具

当用户输入一条任务后，`AgentRunner.stream()` 会触发流式执行。底层 Agent 会根据 system prompt 和上下文消息进行判断：

- 这是不是一个只需要直接回答的问题？
- 是否需要调用工具？
- 如果要调用工具，应该先用哪个？
- 工具返回结果后，是否还要继续调用下一步？

这就是 ReAct 风格 Agent 的典型工作方式：先思考，再行动，再基于结果继续思考。

### 第四步：UI 把黑盒过程变成可观察过程

如果没有 `src/ui/cli.tsx`，这个项目也许依然能工作，但用户会感觉像在面对一个黑盒。

UI 层做了三件很关键的事情：

- 接收用户输入
- 订阅 Agent 的流式输出
- 把当前动作、工具结果、最终回答显示出来

因此，UI 不只是“好看”，它实际上承担了“让 Agent 过程可解释”的作用。

## 哪些模块是主干，哪些是增强项

### 主干模块

以下模块决定了项目能不能跑起来：

- `src/cli.ts`
- `src/agent/agent.ts`
- `src/tools/*.ts`
- `src/ui/cli.tsx`

没有入口，任务进不来；没有 Agent 装配，模型和工具连不起来；没有工具，Agent 只能聊天；没有 UI，交互式体验就不完整。

### 增强模块

以下模块更多是在提升可维护性、性能和安全性：

- `src/utils/config.ts`
- `src/utils/security.ts`
- `src/utils/file-cache.ts`
- `src/agent/prompt.ts`
- `src/agent/state.ts`

这些模块并不总是决定“能不能运行”，但决定“能否长期演进”和“是否更像真实产品”。

## 为什么说它是极简版

当前项目已经具备了一个 Agent 的最小闭环，但还没有走到完整产品化阶段。主要体现在下面几个方面：

### 1. 会话能力还比较轻

虽然项目里有 `sessionId` 和 `MemorySaver`，但 `continue`、`list-sessions` 这些命令还没有真正做完。也就是说，它有“会话”概念，但没有形成完整的会话产品能力。

### 2. 状态设计有预留，但接入不完整

`src/agent/state.ts` 中已经定义了 `AgentState`，包括消息、缓存、执行计划、工具结果等字段，但当前运行主链路还没有把它完整贯穿起来。

这说明设计者已经在为更复杂的 Agent 留扩展位，但目前仍停留在最小实现。

### 3. 安全确认机制还没有闭环

项目里有命令白名单、路径限制，也有 `requireConfirmation` 的配置项，但“真正弹出确认并阻断执行”的主流程还没打通。

### 4. 工具集合还不多

当前工具主要覆盖：

- 文件读写改查
- 命令执行
- 文本搜索
- Git 基础查看与暂存

这已经足够做一个教学型 Agent，但距离“真正强大的工程助手”还有不少空间。

## 推荐的阅读顺序

如果你准备继续往下学，建议按照下面顺序读代码，而不是随机跳：

1. `src/cli.ts`
2. `src/agent/agent.ts`
3. `src/tools/file-tools.ts`
4. `src/tools/exec-tools.ts`
5. `src/ui/cli.tsx`

原因很简单：先看入口，再看装配，再看能力，最后看呈现，这样大脑里的因果关系最清楚。

## 本课小结

这一课最重要的收获，不是记住某个函数名，而是建立一个判断框架：

- 入口在哪里
- 核心装配在哪里
- 能力从哪里来
- 结果如何展示
- 哪些地方已经实现，哪些地方仍是预留

只要这张地图建立起来，后面每一课你都会知道自己在补哪一块认知。
